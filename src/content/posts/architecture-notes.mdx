## 为什么从请求开始

系统设计很容易被画成一张漂亮的框图，但真正能暴露边界的，通常是一次请求从入口到落库再到回传的完整路径。

我习惯先问三个问题：

- 这个请求在哪一步变得不可逆？
- 哪些环节可以重试，哪些环节只能补偿？
- 如果用户说“刚才明明成功了”，我们能否在一分钟内找到证据？

## 边界不是模块名

模块名只能说明代码放在哪里，边界要说明责任在哪里。一个支付回调服务不只是“处理回调”，它还要回答幂等、顺序、证据链和失败补偿。

```ts
type Boundary = {
  owner: string;
  irreversibleAt: string;
  retryPolicy: "safe" | "compensate" | "manual";
  evidence: string[];
};

const paymentCallback: Boundary = {
  owner: "billing",
  irreversibleAt: "ledger_commit",
  retryPolicy: "compensate",
  evidence: ["request_id", "provider_event_id", "ledger_entry_id"]
};
```

## 可观测性要跟着边界走

日志、指标和 tracing 不应该平均撒在每一行代码里。它们更像系统的路标，应该出现在状态变化、跨服务调用和不可逆动作附近。

## 小结

当边界被写清楚，讨论会从“这里是不是要拆服务”转向“谁负责失败后的事实”。这才是架构设计里最省时间的部分。
